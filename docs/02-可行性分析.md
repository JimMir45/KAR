# KAR赛车俱乐部小程序 — 可行性分析报告

> 版本：v1.0 | 日期：2026-03-19
> 依据文档：《01-调研报告.md》《KAR-使用场景说明.md》

---

## 一、项目概述

### 1.1 项目目标

为KAR赛车俱乐部开发一套微信小程序系统，包含C端（用户端）和B端（管理端），覆盖赛道日活动管理、商城、圈速榜、会员体系、财务管理、库存管理、供应商对账、数据看板等全部业务场景。

### 1.2 技术约束

| 约束 | 决策 |
|------|------|
| 前端形态 | 微信小程序（C端+B端各一个） |
| 后端语言 | Go |
| 支付方式 | 微信支付 |
| 建设方式 | 纯自研，不引入SaaS |
| 团队规模 | 1-2人（初期） |

### 2.1 C端功能模块

| **会员体系** | 等级展示、储值充值、余额支付、退费申请 | 可行 | 中 | 滚动12月消费计算需定时任务，储值余额需保证资金安全 |
| **订阅消息** | 报名成功、圈速上榜、会员升级等通知 | 可行 | 低 | 微信订阅消息API成熟，需用户主动订阅 |

|------|---------|-----------|------|------|

| **圈速录入** | 批量录入成绩、自动更新排行榜 | 可行 | 低 | 表格批量输入，保存时触发排行榜更新 |
| **商城管理** | 商品上下架、定价、图片上传 | 可行 | 低 | 标准后台CRUD |
| **订单管理** | 查看订单、退款审批、押金退还 | 可行 | 中 | 退款需调用微信支付退款API，押金冻结/退还需对接微信支付分或押金接口 |
| **会员管理** | 用户列表、消费明细、退费审批、等级规则配置 | 可行 | 中 | 规则引擎相对简单（阶梯门槛+固定折扣） |

| **工单管理** | 到店安装工单、线下工单创建、施工状态流转 | 可行 | 低 | 简单状态机（待处理→施工中→已完成） |
| **财务管理** | 收入自动归集、支出手动录入、固定资产登记、月度报表 | 可行 | 中高 | 收入自动归集需关联订单数据，报表生成需聚合查询 |
| **数据看板** | 周会总览、经营快照、会员概览、库存预警 | 可行 | 中 | 聚合统计查询，数据量小时直接SQL即可，无需BI工具 |
| **供应商管理** | 采购订单、确认发货、对账 | 可行 | 低 | 简单流程，数据量小 |
| **权限体系** | CEO/技师/股东/供应商/超管分角色，不同菜单和操作权限 | 可行 | 中 | RBAC模型，Go生态有Casbin等成熟方案 |

### 2.3 技术难点专项分析

#### 难点1：微信支付（含押金冻结）

| 场景 | 方案 | 可行性 |
|------|------|--------|
| 普通支付（报名费、商品） | 微信支付 JSAPI | 成熟，Go SDK支持完善 |
| 余额支付 | 内部账户扣减，不经过微信支付 | 简单，需保证事务一致性 |
| 退款 | 微信支付退款API | 成熟，支持原路退回 |
| **练习车押金冻结** | 方案A：微信支付分（预授权） | 需申请产品权限，有一定门槛 |
| | 方案B：押金支付接口（V2） | 相对简单，直接收取押金+退还 |
| | 方案C：实际收款，完成后手动退款 | 最简单，但用户体验稍差 |

**建议**：V1版本采用方案C（实际收款+手动退款），功能验证后再升级为支付分预授权。

#### 难点2：滚动12月消费与会员等级

- 需要定时任务（每日/每周）重新计算每个用户近12个月的消费总额
- 数据量预估：活跃用户百级，订单千级，计算量极小
- Go的cron库（如robfig/cron）可轻松实现
- **可行，无技术障碍**

#### 难点3：C端+B端双小程序共享后端

- 两个小程序共用一套Go后端API
- 通过JWT token中的角色信息区分权限
- API路由层做权限校验（中间件）
- **可行，标准做法**

#### 难点4：小程序端富文本编辑

- 微信小程序原生不支持富文本编辑器
- 活动详情的"日程安排"等内容需要格式化展示
- **方案**：B端使用结构化表单（标题+内容块列表）代替富文本，C端用rich-text组件渲染
- **可行，需要产品设计上做适当简化**

## 三、技术可行性总结

### 3.1 Go语言适配性

| 维度 | 评估 |
|------|------|
| Web框架 | Gin / Fiber / Echo 均成熟稳定，Gin社区最大 |
| ORM | GORM 功能完善，支持MySQL/PostgreSQL |
| 微信支付SDK | wechatpay-go（官方）、go-pay（社区）均可用 |
| 微信小程序登录 | 标准HTTP调用，无需SDK |
| 定时任务 | robfig/cron 成熟可靠 |
| 权限管理 | Casbin 是Go生态最成熟的RBAC/ABAC方案 |
| 文件上传 | 对接腾讯云COS或阿里云OSS，SDK完善 |
| 部署 | 单二进制文件，Docker化简单，资源占用低 |
| 性能 | 远超本项目需求（百级并发），无需担忧 |

**结论：Go语言完全适合本项目，且在部署运维方面对小团队非常友好。**

### 3.2 微信小程序适配性

| 维度 | 评估 |
|------|------|
| 功能覆盖 | 场景文档中所有C端和B端功能均可在小程序内实现 |
| 支付集成 | 微信支付在小程序内体验最佳 |
| 订阅消息 | 支持，需用户主动授权 |
| 图片上传 | wx.chooseImage + 上传到后端/OSS |
| 地图导航 | wx.openLocation 可直接调起导航 |
| 分享传播 | 小程序页面分享到朋友圈/微信群，天然契合活动推广 |
| 包大小限制 | 主包2MB，分包合计20MB，需做分包加载 |
| 审核风险 | 商城+支付类小程序需要相关资质（营业执照等） |

| 阶段 | 工作内容 | 预估工时 | 说明 |

| **C端核心功能** | 活动浏览+报名+支付、商城、圈速榜、我的（会员+余额+订单） | 4-6周 | 可分模块并行 |

## 五、风险评估

### 5.1 技术风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|---------|
| 微信支付押金接口权限申请不通过 | 练习车押金功能受限 | 中 | V1先用普通收款+手动退款方式兜底 |
| 小程序审核不通过 | 上线延迟 | 低 | 提前了解审核规则，商城类需提供相关资质 |
| 小程序包体积超限 | 需要重构分包 | 低 | 从一开始就做分包设计 |
| 数据库事务一致性（余额+订单） | 资金安全 | 低 | Go+MySQL事务支持完善，关键操作加分布式锁 |

### 5.2 产品风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|---------|
| 需求蔓延，功能越做越多 | 开发周期拉长 | 高 | 严格分版本迭代，V1只做核心链路 |
| 管理员操作习惯不适应小程序 | B端使用率低 | 中 | B端界面做简化，核心操作控制在3步以内 |
| 用户量太小，系统"空转" | 投入产出不匹配 | 中 | 先小范围试运行，验证核心流程再扩展 |

### 5.3 运营风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|---------|
| 储值余额退费产生纠纷 | 法律合规风险 | 中 | 充值前明确《储值服务协议》，退费规则提前公示 |
| 练习车租赁损伤赔偿纠纷 | 经济损失 | 中 | 线上签署《租赁协议》，押金+协议双重保障 |
| 微信支付费率调整 | 成本上升 | 低 | 商户费率相对稳定，非核心风险 |

## 六、分期迭代建议

### V1（MVP — 核心业务闭环）

**目标**：跑通"发布活动→用户报名支付→活动结束→录入圈速"的完整链路

| 模块 | 范围 |
|------|------|
| C端 | 活动浏览+报名+微信支付、圈速榜（查看）、我的（基本信息+订单） |
| B端 | 活动管理（创建/发布/报名管理）、圈速录入、基础设置 |
| 后端 | 用户系统、活动模块、支付模块、圈速模块 |

**预估周期**：4-6周（2人）

### V2（商城+会员）

| 模块 | 范围 |
|------|------|
| C端 | 商城（浏览+下单+支付）、会员体系（等级+充值+余额支付） |
| B端 | 商城管理（上下架/定价）、会员管理、退费审批、会员规则配置 |

**预估周期**：4-6周（2人）

### V3（运营管理）

| 模块 | 范围 |
|------|------|
| B端 | 库存管理、工单管理、财务管理（收支+资产）、数据看板 |
| 增值服务 | 增值服务绑定活动、练习车租赁+押金 |

**预估周期**：4-6周（2人）

### V4（生态扩展）

| 模块 | 范围 |
|------|------|
| B端 | 供应商模块、月度报表、周会看板 |
| 权限 | 股东只读视角、技师独立视角 |
| 运营 | 订阅消息通知、活动分享海报 |

**预估周期**：3-4周（2人）

## 七、可行性结论

| 维度 | 结论 |
|------|------|
| **技术可行性** | **完全可行** — Go + 微信小程序技术栈成熟，所有功能模块均有明确实现路径，无技术阻断项 |
| **资源可行性** | **可行** — 1-2人团队可在4-6个月内完成全部功能，基础设施月成本150-400元 |
| **经济可行性** | **可行** — 相比SaaS组合方案（年费万元+交易抽成），自研长期成本更低，且数据完全自主可控 |
| **风险可控性** | **可控** — 主要风险为需求蔓延和押金接口权限，均有明确应对方案 |

**总结：项目技术上完全可行，建议分4个版本迭代交付，V1聚焦核心业务闭环，快速上线验证。**
