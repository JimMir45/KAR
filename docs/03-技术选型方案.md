## 二、后端技术选型

### 2.1 Web框架：Gin

| 考量 | 结论 |
|------|------|
| 选择 | **Gin v1.10.x** |
| 淘汰项 | Fiber（基于fasthttp，与标准库不兼容）、Hertz（社区较小）、Echo（Gin社区更大） |

**选择理由**：
- Go生态中采用率最高（48%），社区最大，遇到问题最容易找到解决方案
- 中间件生态最丰富（JWT、CORS、限流、日志等开箱即用）
- 性能满足需求（~34k RPS，远超百级并发场景）
- 学习曲线低，上手快

### 2.2 微信生态SDK

| 功能域 | 选择 | 说明 |
|--------|------|------|
| 小程序登录、订阅消息 | **silenceper/wechat v2** | Go生态最全面的微信SDK，覆盖小程序/公众号/企业微信，Apache 2.0 |
| 微信支付（JSAPI/退款） | **wechatpay-go** | 微信官方SDK，自动签名+验签+证书管理，最可靠 |

**分工**：silenceper/wechat 处理非支付功能（登录、消息），wechatpay-go 处理所有支付相关功能。两者互不冲突，各司其职。

### 2.3 ORM：GORM

| 考量 | 结论 |
|------|------|
| 选择 | **GORM v2** |
| 淘汰项 | Ent（学习曲线高，代码生成模式对小团队增加复杂度）、SQLX（需手写大量SQL，开发效率低）、Bun（社区较小） |

**选择理由**：
- Go ORM采用率最高（~45%），文档和社区资料最丰富
- 事务支持完善（嵌套事务、SavePoint）
- Hook/回调机制，适合订单状态变更等场景的副作用处理
- 关联预加载，减少N+1查询

**数据库迁移**：GORM AutoMigrate（开发阶段）+ **Atlas**（生产环境，自动diff生成迁移文件）

### 2.4 权限管理：Casbin

| 考量 | 结论 |
|------|------|
| 选择 | **Casbin** |
| 模型 | RBAC（基于角色的访问控制） |

**权限模型设计**：

```
角色定义：
- user        → C端普通功能
- shareholder  → user权限 + B端所有模块只读
- technician   → user权限 + B端工单管理/库存管理读写
- ceo          → user权限 + B端大部分模块读写
- super_admin  → 全部权限
- supplier     → B端供应商模块读写
```

Casbin的策略可以存储在数据库中，CEO（兔子）可以通过B端界面调整部分权限配置。

### 2.5 其他后端组件

| 组件 | 选择 | 用途 |
|------|------|------|
| 配置管理 | **Viper** | 支持YAML/ENV/远程配置，Go生态标准 |
| 日志 | **Zap** | 高性能结构化日志，uber出品 |
| 定时任务 | **robfig/cron v3** | 滚动12月消费计算、会员等级刷新 |
| 参数校验 | **go-playground/validator** | Gin默认集成，结构体tag校验 |
| API文档 | **swaggo/swag** | 从注释自动生成Swagger文档 |
| 错误处理 | **pkg/errors 或标准库 errors** | 错误包装和堆栈追踪 |

## 三、前端技术选型

### 3.1 小程序框架：Taro 4.x + React

| 考量 | 结论 |
|------|------|
| 选择 | **Taro 4.x + React + TypeScript** |
| 淘汰项 | uni-app（绑定HBuilderX、Vue生态）、微信原生（代码复用率低） |

**选择理由**：
- React + TypeScript 生态更大，类型安全性更好
- Taro 4.x 支持 Vite 编译，开发体验好
- 不绑定特定IDE（VS Code即可）
- C端和B端可共享大量组件和工具代码
- 如后续需要H5端或鸿蒙端，Taro跨端能力强

### 3.2 UI组件库：NutUI React

| 考量 | 结论 |
|------|------|
| 选择 | **@nutui/nutui-react-taro** |

**选择理由**：
- 京东出品，与Taro官方合作，兼容性最好
- 70+组件，覆盖表单、列表、弹窗、导航等常用场景
- 支持按需加载，不增加包体积
- TypeScript编写，类型提示完善

### 3.3 Monorepo架构：pnpm Workspaces

| 考量 | 结论 |
|------|------|
| 选择 | **pnpm Workspaces** |
| 淘汰项 | Turborepo（项目规模不大，全量构建<30秒，不需要构建缓存） |

**项目结构**：

### 3.4 状态管理：Zustand

| 考量 | 结论 |
|------|------|
| 选择 | **Zustand** |
| 淘汰项 | Redux（过重）、Jotai（原子化模型对本项目过于细粒度） |

**选择理由**：轻量、API简洁、TypeScript友好、无Provider包裹。

## 四、数据库选型

### 4.1 主数据库：PostgreSQL

| 考量 | 结论 |
|------|------|
| 选择 | **PostgreSQL 16+** |
| 淘汰项 | MySQL（JSON支持较弱、长期演进空间较小） |

**选择理由**：
- JSONB支持完善 — 活动详情的日程安排、商品规格等半结构化数据可用JSONB存储，支持GIN索引
- 事务与数据完整性更严格 — 订单、余额等资金相关操作更可靠
- 2025年Stack Overflow调研中已超越MySQL成为专业开发者最常用数据库
- Go生态支持完善（GORM通过pgx驱动，性能优秀）
- 许可证无限制（BSD类），MySQL为GPL

### 4.2 缓存：Redis（V2阶段引入）

| 阶段 | 策略 |
|------|------|
| **V1（MVP）** | **不引入Redis**。JWT做会话、数据库行锁做库存、内存级限流 |
| **V2（商城+会员）** | 引入Redis，用于圈速排行榜（Sorted Set）、热点商品缓存 |
| **V3+** | 扩展到库存扣减锁、限流、会话管理 |

**理由**：初期用户量小（百级），减少运维组件，Go单实例性能足够。

## 五、基础设施选型

### 5.1 服务器：腾讯云轻量应用服务器

| 考量 | 结论 |
|------|------|
| 选择 | **腾讯云轻量应用服务器 2核4G** |
| 淘汰项 | ECS/CVM（贵）、云托管（需容器知识）、Serverless（调试复杂） |

**选择理由**：
- 性价比最高，套餐价约70-150元/月
- 预装Docker CE镜像，开箱即用
- 足以支撑早期流量（Go + PostgreSQL单机可轻松处理数千QPS）
- 后续业务增长可迁移到CVM

### 5.2 对象存储：腾讯云COS + 客户端直传

| 考量 | 结论 |
|------|------|
| 选择 | **腾讯云COS** |
| 上传方式 | **客户端直传**（小程序端获取临时密钥 → 直传COS） |
| 淘汰项 | 阿里云OSS（免费额度仅5GB，COS为50GB）、MinIO（需运维） |

**上传流程**：
1. 小程序调用后端接口获取COS临时密钥（STS）
2. 小程序使用wx.uploadFile直传COS
3. COS返回文件URL，小程序回传给后端入库

**优势**：不占后端带宽、速度快、与微信生态天然集成、免费额度大。

### 5.3 反向代理/HTTPS：Caddy

| 考量 | 结论 |
|------|------|
| 选择 | **Caddy v2** |
| 淘汰项 | Nginx（需手动配置SSL证书和续期） |

**选择理由**：
- 内建自动HTTPS — 自动申请Let's Encrypt证书并续期，零配置
- 配置文件极简（Caddyfile几行即可）
- 性能足够，适合单机部署
- 微信小程序要求HTTPS + 443端口，Caddy天然满足

### 5.4 部署方案：Docker Compose

```yaml
# docker-compose.yaml 结构
services:
  app:        # Go后端（多阶段构建，镜像<20MB）
  postgres:   # PostgreSQL 16
  caddy:      # 反向代理 + 自动HTTPS
  # redis:    # V2阶段启用
```

**部署流程（MVP阶段）**：
1. 本地 `docker build` 构建Go镜像
2. `docker save` 导出 → scp到服务器
3. 服务器 `docker load` + `docker compose up -d`

**后续升级**：接入GitHub Actions自动构建部署。

### 5.5 域名与备案

| 项目 | 方案 | 备注 |
|------|------|------|
| 域名注册 | 腾讯云 | 方便备案 |
| ICP备案 | 腾讯云备案小程序提交 | **7-20个工作日，立即启动** |
| SSL证书 | Caddy自动管理（Let's Encrypt） | 免费，90天自动续期 |
| 小程序域名配置 | 在微信后台配置request/upload/download合法域名 | 备案通过后配置 |

## 八、关键技术决策记录

| # | 决策 | 理由 |
|---|------|------|
| 1 | Go + Gin 作为后端 | 性能好、部署简单（单二进制）、小团队运维友好 |
| 2 | Taro + React 作为前端 | 跨端能力强、不绑定IDE、React/TS生态大 |
| 3 | PostgreSQL 作为主库 | JSONB支持好、事务严格、长期演进空间大 |
| 4 | V1不引入Redis | 减少运维组件，初期数据量小不需要 |
| 5 | 腾讯云COS客户端直传 | 不占后端带宽、免费额度50GB、微信生态集成 |
| 6 | Caddy替代Nginx | 自动HTTPS零配置、Caddyfile极简 |
