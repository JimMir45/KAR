## 一、产品概述

### 1.1 产品定位

KAR赛车俱乐部微信小程序，为俱乐部提供线上化运营工具，覆盖赛道日活动管理、商城销售、圈速排行、会员运营、财务管理等核心业务。

### 1.2 产品形态

| 平台 | 形态 | 使用者 |
|------|------|--------|
| C端（用户端） | 微信小程序 | 车友、会员、股东 |
| B端（管理端） | 微信小程序（V1）| CEO、技师、股东、供应商、超级管理员 |

两端共享同一Go后端，通过角色权限区分功能。

### 1.3 迭代版本规划

| 版本 | 范围 | 目标 |
|------|------|------|
| **V1** | 赛道日活动（发布/报名/支付）+ 圈速榜 + 基础用户系统 | 跑通核心业务闭环 |
| **V2** | 商城 + 会员体系（等级/充值/余额支付/退费） | 建立用户粘性 |
| **V3** | 库存管理 + 工单管理 + 财务管理 + 数据看板 | 完善运营管理 |
| **V4** | 供应商模块 + 股东视角 + 订阅消息 + 分享海报 | 生态扩展 |

## 二、角色与权限

### 2.1 角色定义

| 角色 | 标识 | 说明 |
|------|------|------|
| **用户** | user | 普通车友，C端全部功能 |
| **CEO** | ceo | 二楼运营管理员（兔子），C端用户功能 + B端运营管理 |
| **技师** | technician | 一楼技师管理员（小黑），C端用户功能 + B端库存/工单管理 |
| **股东** | shareholder | 投资人（元宝），C端用户功能（默认最高等级+固定折扣）+ B端所有数据只读 |
| **供应商** | supplier | 合作商家（赛轮张经理），仅B端供应商模块 |
| **超级管理员** | super_admin | 研发/运维，全部权限 |

### 2.3 股东特殊规则

- 股东账号默认为**最高会员等级**（当前为黑卡），不受滚动消费金额影响
- 享受CEO配置的**股东固定折扣**（独立于会员等级折扣）
- B端所有模块仅有**只读权限**（所有操作按钮灰色不可点击）

## 四、C端功能需求

### 4.1 首页

**页面元素**：
- 顶部Banner轮播（最新活动封面/赞助商）
- "即将开始的赛道日"活动卡片列表
  - 卡片信息：活动名称、日期、赛道、价格、剩余名额/总名额、状态标签（报名中/已截止/已结束）
- "历史活动"入口链接
- 底部Tab导航：首页 | 商城 | 圈速榜 | 我的

**业务规则**：
- 活动按日期倒序排列，状态为"报名中"的置顶
- 已结束的活动默认不在首页展示，进入"历史活动"查看
- 活动卡片可长按分享到微信群/朋友圈

### 4.2 活动详情页

**页面元素**：
- 封面图（顶部）
- 赞助商Logo（封面图上方角落）
- 活动基本信息：名称、日期、赛道、价格、剩余名额
- 日程安排列表（时间+内容，结构化展示）
- 赛制说明
- 冠军奖品
- 注意事项列表
- 场地保险提示文字
- 赛道地址（点击调起导航）
- 赛道布局图（可放大查看）
- 旗语说明图（可放大查看）
- 增值服务卡片列表（横向滚动）
  - 每张卡片：服务名称、价格、剩余名额
  - 点击进入增值服务详情
- 底部固定栏："立即报名 ¥XXX"按钮

**业务规则**：
- 报名截止后按钮变为灰色"报名已截止"
- 活动结束后展示"活动已结束"，可查看当期活动榜
- 已报名用户显示"已报名"状态，不可重复报名
- 增值服务卡片只展示CEO为本期活动配置的服务

### 4.3 报名流程

**步骤1 — 填写信息**：
- 姓名（首次填写，后续自动带入，可修改）
- 手机号（微信一键获取）
- 车牌号（必填，说明"用于报备场地门卫"）
- 车型（必填，如"高尔夫GTI"）

**步骤2 — 费用确认**：
- 报名费原价
- 会员折扣（如有）：显示 ~~¥750~~ → ¥712.5（银卡95折）
- 合计金额
- 支付方式选择：微信支付 / 储值余额（显示当前余额）

**步骤3 — 支付**：
- 微信支付：调起微信支付弹窗
- 余额支付：直接扣减余额，无需弹窗

**步骤4 — 支付成功页**：
- 报名成功提示
- 活动名称、日期、地点
- 当前会员等级权益提醒（如金卡可免费使用头盔）
- "查看报名详情"按钮

**订阅消息**：支付成功后发送报名成功通知。

**业务规则**：
- 名额已满时不可报名
- 报名截止后不可报名
- 同一用户可为他人代报名（填写他人信息），每次报名占1个名额
- 报名费按报名人（下单用户）的会员等级计算折扣
- 余额不足时提示"余额不足，请先充值或选择微信支付"

### 4.4 增值服务购买

**页面**：增值服务详情页

**页面元素**：
- 服务名称、价格、详情描述
- "所属活动：XXX"（自动绑定，不可更改）
- 库存/名额状态
- 如需签署协议（练习车），展示协议全文 + "我已阅读并同意"勾选框
- 费用确认（含押金，如有）
- 支付按钮

**业务规则**：
- 增值服务必须绑定赛道日活动购买，不可在商城独立购买
- 增值服务不要求用户先报名该活动（可单独购买）
- 练习车租赁需签署协议后才可支付
- 练习车押金：V1版本为实际收款，活动结束后CEO手动操作退款
- 会员折扣适用于增值服务价格（不含押金部分）

## 六、核心业务规则汇总

### 6.1 会员等级计算

```
触发时机：每日定时任务（凌晨执行）
计算逻辑：
1. 遍历所有用户
2. 计算每个用户近12个月（滚动窗口）的实际消费总额
   - 消费 = 所有已支付订单的实付金额（不含充值）
   - 已退款的金额不计入
3. 根据消费总额匹配等级门槛（从高到低匹配）
4. 股东账号跳过计算，始终为最高等级
5. 如等级发生变化，更新用户等级并发送通知
```

### 6.2 折扣计算

```
折扣优先级：
1. 股东：使用股东固定折扣
2. 非股东：使用当前会员等级折扣
3. 无等级（车友Lv.0）：无折扣（原价）

折扣适用范围：
- 赛道日报名费 ✓
- 商城商品 ✓
- 增值服务 ✓
- 押金 ✗（不参与折扣）
- 充值 ✗（不参与折扣）
```

### 6.3 订单状态机

```
普通订单（报名/商城/增值服务）：
  待支付 → 已支付 → 已完成
  待支付 → 已取消（超时未支付）
  已支付 → 申请退款 → 已退款
  已支付 → 待到店 → 施工中 → 已完成（到店安装类）

含押金订单（练习车）：
  待支付 → 已支付（含押金冻结）→ 已完成（押金退还）
  已支付 → 已完成（押金扣除，有损伤）

充值订单：
  待支付 → 已支付（余额增加）
```

### 6.4 退款规则

```
赛道日报名退款（阶梯规则，CEO可配置）：
  活动前7天以上 → 全额退款（100%）
  活动前3-6天   → 退50%
  活动前3天以内 → 不可退款（0%）

储值余额退费（CEO可配置）：
  扣除X%手续费后退还
  CEO可在审批时调整退费比例

退款渠道：
  微信支付 → 原路退回微信
  余额支付 → 退回储值余额
```

### 6.5 库存扣减

```
自动扣减（工单完成时）：
  技师点击"施工完成" → 扣减工单中列出的配件库存

手动扣减（内部消耗）：
  技师手动创建出库记录 → 扣减库存

库存增加（入库时）：
  技师手动创建入库记录 → 增加库存 → 同步C端商城显示
```

### 6.6 财务自动归集

```
以下收入自动计入财务系统：
- 赛道日报名费（支付成功时）
- 商城订单支付（支付成功时）
- 增值服务购买（支付成功时）
- 充值收入（支付成功时）

以下支出自动计入：
- 配件入库采购（入库操作时，按进货单价×数量）

手动录入：
- 场地费、奖品、餐饮、燃油等线下支出由CEO手动录入
- 退款支出（退款审批通过时自动记录）
```

## 七、非功能性需求

### 7.1 性能

| 指标 | 要求 |
|------|------|
| API响应时间 | P95 < 500ms |
| 页面首屏加载 | < 2s |
| 并发支持 | 100+同时在线（初期） |

### 7.2 安全

| 项目 | 要求 |
|------|------|
| 认证 | JWT + 微信登录，token有效期7天 |
| 权限 | 所有B端接口必须校验角色权限 |
| 支付安全 | 支付相关操作全部在服务端完成，禁止前端暴露密钥 |
| 数据安全 | 用户手机号脱敏展示（中间4位*号） |
| 资金安全 | 余额变动必须在数据库事务内完成，使用行锁防止并发问题 |
| SQL注入 | 使用ORM参数化查询，禁止拼接SQL |
| XSS | 用户输入内容在展示前转义 |

### 7.3 可用性

| 项目 | 要求 |
|------|------|
| 数据备份 | 每日自动备份数据库 |
| 日志 | 关键操作记录审计日志（支付、退款、库存变动、权限变更） |
| 监控 | 服务健康检查接口，异常时通知 |
